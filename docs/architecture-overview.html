Emergency Guidance Agent — Architecture Overview

Multimodal coaching for lay rescuers: browser audio and video stream to a backend session, a conversational model drives guidance, and a deterministic workflow engine advances CPR-oriented steps. This page summarizes how the repositories and integrations fit together.

End-to-end flow

Default product path: CPR Copilot (React) talks to the Python FastAPI backend (VITE_API_URL, usually http://localhost:8000). One coaching session follows the steps below; the sequence diagram shows the same story in message order.

1

Create session

Browser POST /session. Server allocates id and in-memory SessionState (intake).

→
2

Open realtime link

Pipecat client opens WebSocket to /ws/stream/<session_id> with protobuf framing. Mic is 16 kHz; camera preview stays local for JPEG encode.

→
3

Boot pipeline

Server builds Pipecat pipeline: websocket transport in, GeminiLiveLLMService, WorkflowProcessor, transport out. Gemini Live connects with API key and step-aware system prompt.

→
4

Opening guidance

Seeded user message in context triggers the model to speak first (short intro + “what can I help with”). TTS streams back through Pipecat to the browser.

→
5

Live coaching loop

Continuous audio in; periodic JPEG frames as raw messages. Gemini transcribes, reasons, and replies with speech. Client shows transcripts and instruction text in the UI.

→
6

Workflow FSM

On user transcription and completed model text, heuristics build ModelInterpretation. evaluate / apply_decision updates step and session; on advance, step context is pushed back into Gemini via TextFrame.

→
7

Teardown

WebSocket closes (user ends session or error). Runner stops; end_session removes server state; client releases camera and mic tracks.

Data plane

Audio and images move through the Pipecat websocket; Gemini Live handles perception and generation; TTS audio returns on the same pipe.

Control plane

Session row in session_manager, workflow step, and optional RTVI-style client messages (e.g. done / repeat) govern what instruction set applies—not a second HTTP API per turn.

Repository layout

  • cpr-copilot — React (Vite) UI: camera preview, Pipecat client for WebSocket + mic, optional browser speech hooks, transcript and instruction panels.
  • backend — Python FastAPI server: POST /session, WebSocket /ws/stream/{session_id}, in-memory session store, Pipecat pipeline to Google Gemini Live.
  • apps/api — Node (Fastify) alternate API: POST /session, WebSocket /ws/:sessionId with JSON ClientToBackendEvent messages and a stub Gemini orchestrator (no API key) for wiring tests.
  • packages/shared — TypeScript types for session state, workflow steps, model interpretation shape, and PRD-aligned client event names (consumed by apps/api).

Primary runtime path Production demo

The CPR Copilot app defaults to VITE_API_URL http://localhost:8000, so the live stack is the Python backend. See Architecture diagrams above for the visual overview.

Integration details (primary path)

Integration Role
@pipecat-ai/client-js Connects the browser to the backend WebSocket; handles audio in/out sample rates (16 kHz in, 24 kHz playback) and bot/user transcript events.
ProtobufFrameSerializerCompat Frontend serializer aligned with Pipecat’s protobuf framing for the FastAPI websocket transport.
GeminiLiveLLMService Streams bidirectional audio and reasoning; model from env GEMINI_LIVE_MODEL (default preview native-audio id in code). Requires GEMINI_API_KEY.
WorkflowProcessor Subscribes to TranscriptionFrame and completed LLM text; builds ModelInterpretation, runs Python workflow_engine, updates session, injects TextFrame step context when the FSM advances.
Raw video frames Canvas JPEG blobs sent as Pipecat raw messages for the bot path; complements mic audio for multimodal coaching.
Parallel module: backend/gemini_session.py wraps the lower-level genai.Client.aio.live.connect API for experiments (e.g. explicit JPEG/PCM sends). The main product path for the web app is pipeline.py + Pipecat’s GeminiLiveLLMService.

Alternate path TypeScript / stub

apps/api implements the same conceptual flow with Fastify: create session, upgrade WebSocket, accept JSON envelopes matching @emergency-guidance/shared (speech.transcript, media.frame, user.done, etc.). The createStubGeminiOrchestrator() returns heuristic interpretations so workflow_engine.ts can run without calling Google. Useful for typecheck-only workflows or future swap-in of a real TS Gemini client.

Workflow steps (shared concept)

Both backends model the same high-level graph: intake → escalation → see_patient → start_compressions → continue_cpr → complete. Python and TypeScript engines encode slightly different rule sets; both advance state from model signals and (where wired) explicit user actions.
